01

Project Description

The aim of this project is to consolidate everything covered across the course by building a complete, production-structured Angular application connected to a real backend API. Rather than practising concepts in isolation, you will apply them together — the way they are used in professional frontend development — and observe how each piece of the framework serves a distinct purpose within a working system.

The application is a Library Management System with two user roles: a regular user who can browse the catalogue, borrow books, track their loans, and pay fines; and a librarian who can add and edit books, manage all user accounts, and view dashboard statistics. You are given the folder structure, the name of every file you must create, and the behaviour each piece must implement — but no implementation code. The entire codebase is yours to write.

API Server

Start the provided backend with npm start in the server folder. It runs on http://localhost:3001. All data resets on restart — that's fine. Use the Student Quick Reference section of the API docs to understand the workflow.

What to Build — Module by Module

Each card below tells you what to create and what it must do. The syllabus reference next to each card tells you which course module the concept belongs to.

This structure is intentional. Each card does not just describe a technical requirement — it demonstrates where and why that concept exists in a real application. By the end, you will have a clear mental model of how components, services, pipes, forms, routing, and HTTP fit together in production Angular code, not just as academic exercises but as the specific design decisions that make large-scale frontends maintainable.

Card A Components & Templates Syllabus Mod 2–3

Every component listed in the folder structure must be created as a standalone component.

  • app.component — Root shell. Contains the <router-outlet> and a navigation bar. The navbar shows different links depending on whether the user is logged in and whether they are a librarian. Use property binding and string interpolation to display the logged-in user's name.
  • book-card.component — Pass book here and emit when borrowed button is clicked. Uses class binding to visually indicate when a book has zero available copies.
  • loading-spinner.component — Conditionally rendered in parent templates while HTTP calls are in progress. No external library — implement with pure CSS animation.
  • confirm-dialog.component — Accepts a @Input() message string and emits @Output() confirmed and @Output() cancelled. You may use this for actions that required user input
  • All templates must use the safe navigation operator (?.) wherever a property could be null (e.g. user?.fullName).
@Component @Input / @Output lifecycle hooks ngOnInit · ngOnDestroy
Card B Directives Syllabus Mod 4

You must use built-in structural and attribute directives throughout your templates, and implement the two custom directives below.

  • Use appropriate structural directives to conditionally display the "Borrow" button only when it makes sense (consider availability and whether the user already has the book).
  • Use repeating directives with trackBy on every list rendering in the application.
  • Use attribute directives to apply different visual styles to issue rows based on their status.
  • highlight.directive — Custom attribute directive. Apply to book-card elements. It should produce a visual highlight effect on hover. The highlight colour should be configurable by the consuming template.
  • role.directive — Custom structural directive. It conditionally renders its host element based on whether the current user's role matches a given role string. Think about how Angular's built-in structural directives work internally and replicate that pattern.
*ngIf / @if *ngFor / @for ngClass · ngStyle @Directive @HostListener
Card C Services & Dependency Injection Syllabus Mod 6

All business logic and state must live in services, not in components. Every service must be decorated with @Injectable({ providedIn: 'root' }).

  • auth.service — Manages the full authentication lifecycle (login, register, logout). Must persist the session token client-side and expose a reactive stream of the current user that any component can subscribe to. Provide convenience methods for checking auth state and user role. The session should survive a page refresh.
  • book.service — Wraps all book-related HTTP calls. Must return observables from every method. Consult the API Reference (Section 04) to determine which endpoints to call for listing, searching, filtering by category, and CRUD operations. Apply appropriate error-handling operators.
  • issue.service — Handles borrowing, returning, and renewing books. Refer to the Issues section of the API Reference for the correct endpoints and payloads.
  • fine.service — Handles retrieving a user's fines and paying them. Refer to the Fines section of the API Reference for the correct endpoints.
@Injectable BehaviorSubject providedIn: 'root' singleton
Card D Pipes & Data Transformation Syllabus Mod 7

You must use built-in Angular pipes in your templates and implement the three custom pipes below.

  • Use appropriate built-in Angular pipes for formatting dates, currency values, and text casing throughout your templates.
  • days-until.pipe — Transforms a date string into a human-friendly relative label (e.g. how many days remain, or how overdue it is). Pure pipe.
  • overdue-status.pipe — Takes an Issue object and derives its display status. Used alongside attribute directives to style status badges. Pure pipe.
  • truncate.pipe — Shortens long strings beyond a configurable character limit, appending an ellipsis. Used for book descriptions in the card view. Pure pipe.
@Pipe PipeTransform pure pipe date · currency · titlecase parameterised pipe
Card E Component Communication Syllabus Mod 8

Data must flow correctly between components using the appropriate mechanism for each situation.

  • Parent → Child: book-list passes each Book object down to book-card via @Input().
  • Child → Parent: book-card emits a borrowClicked event via @Output() + EventEmitter. book-list handles it and calls issue.service.borrowBook().
  • Unrelated components via service: auth.service's currentUser$ BehaviorSubject is subscribed to in both app.component (to update the navbar) and my-books.component (to personalise the greeting). They must not talk to each other directly.
  • ViewChild: In book-list.component, use @ViewChild to get a reference to a search <input> element and focus it programmatically when the component loads (ngAfterViewInit).
  • Content projection: The confirm-dialog must use <ng-content> to project a custom message body passed from the parent.
@Input · @Output EventEmitter BehaviorSubject @ViewChild ng-content
Card F Template-Driven & Reactive Forms Syllabus Mod 9–10

You will use one form of each type. Both forms must show validation error messages to the user. Because this app uses standalone components, add FormsModule or ReactiveFormsModule directly to each component's own imports: [] array — there is no AppModule to put them in.

  • login.component — Template-Driven Form. Fields: username (required, minlength 3) and password (required, minlength 6). Use [(ngModel)] and ngForm. Show an error message under each field when it is touched and invalid. Disable the submit button when the form is invalid.
  • register.component — Template-Driven Form. Fields: username, password, email (required + valid format), fullName (required). Same validation error display pattern as login. On submit call auth.service.register().
  • book-form.component — Reactive Form (librarian only). Fields: title (required), body (required — this is the book description), isbn, category, publishedYear (number), totalCopies (number). Use FormBuilder and FormGroup. Add Validators.required to title and body, and Validators.min(1) on copies. The form must support both creating a new book and editing an existing one — it should patchValue when an existing book is passed in via @Input(). Note: the backend has no author or publisher fields — do not include them.
imports: [FormsModule] ngModel · ngForm imports: [ReactiveFormsModule] FormBuilder Validators patchValue
Card G Routing & Navigation Syllabus Mod 11

Define all routes in app.routes.ts. All feature routes must be lazy-loaded. Wire the router in app.config.ts using provideRouter(routes) — not RouterModule.forRoot(). Import RouterOutlet in app.component and RouterLink / RouterLinkActive directly in whichever components use them.

  • Define routes for: authentication pages, the public book catalogue, individual book detail, the user's personal loans page, and the librarian management views (dashboard, books, users). Add a wildcard route that catches everything else.
  • All feature areas must be lazy-loaded — the main route file should not directly import any feature components. Instead, delegate to each feature's own route file.
  • The book detail page receives a book identifier via the URL. Use a reactive approach to read this parameter (not a one-time snapshot) and load the corresponding book.
  • After a successful login, navigate the user away from the login page programmatically.
  • The navigation bar should use Angular's router directives for links and visually highlight the currently active route.
  • Apply the appropriate guards (see Card I) to routes that require authentication or a specific role.
RouterOutlet · RouterLink loadComponent loadChildren *.routes.ts ActivatedRoute paramMap Router.navigate routerLink
Card H HTTP Client & Observables Syllabus Mod 12

All HTTP communication must go through the services you built in Card C using HttpClient. Components subscribe to these observables and update the UI. Register HTTP with provideHttpClient(withInterceptors([...])) in app.config.ts — do not use HttpClientModule.

  • In book.service, the search() method must accept a query string. Build a HttpParams object with the key q and pass it to the GET request to /api/books/search — do not manually concatenate query strings. Note: the search endpoint does not support category filtering; for that, call /api/books/category/:category separately.
  • In book-list.component, pipe the search input event through debounceTime(300) and distinctUntilChanged() before calling the service. This requires creating an observable from the input event using fromEvent.
  • In every component that subscribes to an observable, unsubscribe in ngOnDestroy to prevent memory leaks. Use either a Subject + takeUntil pattern or the async pipe in the template.
  • Use the async pipe in at least one template (e.g. the books list) to display data from an observable without manually subscribing.
  • Use catchError and retry(1) operators in book.service.getAll() to handle transient failures gracefully.
HttpClient HttpParams Observable async pipe debounceTime catchError · retry takeUntil
Card I Authentication, Guards & Interceptors Syllabus Mod 13

Implement two interceptors and two guards. All four must be functional — they are part of the marking criteria. Register both interceptors in app.config.ts inside provideHttpClient(withInterceptors([authInterceptor, errorInterceptor])) — there is no HTTP_INTERCEPTORS token needed in standalone.

  • auth.interceptor — Ensures that authenticated API requests carry the user's token in the appropriate HTTP header. Think carefully about which requests should not have a token attached.
  • error.interceptor — A global response handler. Authentication failures (think: what status codes signal "you're not allowed here"?) should log the user out and redirect. Server errors should be logged and re-thrown so individual components can react if needed.
  • auth.guard — Prevents unauthenticated users from reaching protected routes. Unauthenticated visitors should be redirected to the login page.
  • librarian.guard — Prevents non-librarian users from reaching librarian-only routes, even if they are logged in. Non-librarians should be redirected elsewhere.
HttpInterceptorFn withInterceptors cloneRequest catchError CanActivateFn JWT + localStorage
Card J Lazy Loading & Best Practices Syllabus Mod 14

Performance and code quality requirements. These are checked during code review, not just by running the app.

  • Lazy loading: app.routes.ts uses loadChildren to point to each feature's *.routes.ts file. Inside feature route files, use loadComponent to load the individual components. Verify in DevTools Network tab — feature bundles must only load when the route is first visited.
  • OnPush change detection: Apply ChangeDetectionStrategy.OnPush to book-card.component and loading-spinner.component. These are pure display components that should only re-render when their inputs change.
  • TrackBy: Every *ngFor / @for loop in the app must use a trackBy function (or track expression) that tracks by the item's id property.
  • No memory leaks: Use the takeUntil(this.destroy$) pattern with a Subject in every component that manually subscribes to an observable. The subject must be completed in ngOnDestroy.
  • TypeScript types: No use of any allowed. All HTTP responses must be typed using the interfaces in app/models/.
loadChildren loadComponent OnPush trackBy / track takeUntil · destroy$ TypeScript interfaces

Important — Fine & Renewal Rules

The grace period is 1 day. After that, the rate is $0.50/day, up to a maximum of $20 per book. The fineAmount field on the issue object is calculated automatically by the server on return. Renewals extend the due date by 7 days (not 14) and are capped at 2 per issue. Overdue books cannot be renewed.

// Helper — compute these client-side, they are NOT returned by the server
export function getDaysUntilDue(issue: Issue): number {
  const now = new Date();
  const due = new Date(issue.dueDate);
  return Math.ceil((due.getTime() - now.getTime()) / (1000 * 60 * 60 * 24));
}

export function isOverdue(issue: Issue): boolean {
  return issue.status === 'overdue';
}

API response wrappers & key field names

Some endpoints return wrapper objects rather than bare arrays. Type these inline in your service methods rather than creating separate interfaces. The login endpoint returns Observable<{ message: string, token: string, user: User }>. The issue borrow endpoint returns Observable<{ message: string, issue: Issue, book: { id, title } }>. The return endpoint returns Observable<{ message: string, issue: Issue, fine: number, hasFine: boolean }>. Always use GET /api/issues (or /api/issues/user/:userId) for a user's loans — there is no /api/issues/my endpoint. For fines, use GET /api/fines/user/:userId — there is no /api/fines/my. Remember: status is 'issued' | 'overdue' | 'returned', the fine amount field is fineAmount, and the renewal count field is renewalCount.

Rules & Submission

No-Code Rule

The task description above gives you the structure and the behaviour to implement. Do not paste solutions from the internet or use AI tools to write your implementation code. Understanding how each piece works is the entire point. During the code review, you may be asked to explain any part of your submission.

What to Submit

A single ZIP of your Angular project folder (excluding node_modules). Include a short README.md that explains how to run the project and lists any features you did not complete and why.

Common Pitfalls to Avoid

Memory leaks — always unsubscribe. Logic in templates — keep templates declarative; use pipes or component methods instead. Calling HTTP in components — all HTTP must go through services. Hardcoded tokens — always read from localStorage via auth.service. Missing OnPush on display components will lose marks even if the app works.
